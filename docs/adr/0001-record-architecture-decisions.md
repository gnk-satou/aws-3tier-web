# ADR-0001: アーキテクチャ上の意思決定を ADR として記録する

- Status: Accepted
- Date: 2026-07-03

## Context

段階構築するポートフォリオでは「なぜその構成にしたか」を後から説明できることが重要。
コードだけでは設計判断の理由が残らない。

## Decision

重要な設計判断は Michael Nygard 形式の軽量 ADR として `docs/adr/` に連番で記録する。
1 ADR = 1 判断。決定を覆す場合は新しい ADR で Supersede する。

## Consequences

- 面接やレビューで設計意図を根拠付きで説明できる
- 判断の変更履歴が Git 上に残る
- 書くコストが増えるが、1 件あたり短く保つことで許容する
